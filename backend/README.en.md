## Description

The backend exposes an API with FastAPI and persists the data in PostgreSQL. Its main responsibilities are:

1. Record and query the deauthentication alerts captured by the ESP32 nodes.
2. Expose the data through REST endpoints and a real-time WebSocket channel.
3. Authenticate users (JWT) and services (machine-to-machine credential for the processing layer).

For the full system context (architecture, other layers, academic evidence), see the [main README](../README.en.md). The [main README also documents the Docker deployment](../README.en.md#getting-started), which brings this backend up together with PostgreSQL and the frontend.

## Environment configuration

### Prerequisites

- Python 3.12, the version validated by the backend's Docker image.
- Git.
- A running PostgreSQL instance (local or via the Docker Compose in the [main README](../README.en.md)).

## Installation and startup

### 1. Clone the repository and enter the backend

```bash
git clone https://github.com/Eberth-sys/Deauth-Alert-WiFi-IoT-System.git
cd Deauth-Alert-WiFi-IoT-System/backend
```

### 2. Create and activate a virtual environment

```bash
python -m venv .venv
source .venv/bin/activate      # Linux/macOS
.venv\Scripts\activate         # Windows
```

### 3. Install dependencies

```bash
pip install --upgrade pip
pip install -r src/requirements.txt
```

### 4. Configure the environment variables

```bash
cp src/.env.example src/.env
```

At a minimum, fill in the following in `src/.env`:

| Variable | Description |
| --- | --- |
| `PG_DB`, `PG_USER`, `PG_PASSWORD`, `PG_HOST`, `PG_PORT` | PostgreSQL connection. The backend derives `DATABASE_URL` from these variables; you do not need to define it. |
| `JWT_SECRET_KEY` | Key used to sign the JWTs. It must be 32 characters or longer: the backend fails to start if it is shorter. |
| `SERVICE_API_KEY` | Credential used by the processing layer to update the state of the ESP32 nodes. |
| `CORS_ORIGINS` | Allowed origins, comma-separated and without spaces (for example, `http://localhost:5173,http://localhost:8080`). |
| `LOGS_DIR` | Folder where the log files are stored. |
| `FRONTEND_URL` | Frontend URL, used in links generated by the backend. |

Keep `src/.env` out of version control (it is already in `.gitignore`).

### 5. Start the server

```bash
uvicorn src.main:app --reload --host 0.0.0.0 --port 8000
```

This is the same command (without `--reload`) that the backend's Docker image uses.

### 6. View the interactive documentation

With the server running, open [http://localhost:8000/docs](http://localhost:8000/docs) (Swagger UI) to explore and try out the endpoints.

## REST and WebSocket endpoints

`get_current_user` requires a valid user JWT (`Authorization: Bearer <token>` header). `get_current_admin_user` additionally requires the administrator role. `verify_service_key` requires the `X-API-Key` header set to the value of `SERVICE_API_KEY`.

### Authentication (`/auth`)

| Method | Route | Auth | Description |
| --- | --- | --- | --- |
| POST | `/auth/register` | None | Registers a new user. |
| POST | `/auth/login` | None | Logs in and returns a JWT. |
| GET | `/auth/me` | User | Returns the authenticated user's data. |
| POST | `/auth/forgot-password` | None | Requests a password reset. |
| POST | `/auth/reset-password` | None (reset token in the body) | Resets the password using the token received. |

### Alerts and nodes

| Method | Route | Auth | Description |
| --- | --- | --- | --- |
| GET | `/alerts` | User | Lists the recorded alerts. |
| GET | `/alerts-summary` | User | Aggregated alert statistics. |
| GET | `/esp32-nodes` | User | Lists the state of the ESP32 nodes. |
| POST | `/esp32-nodes/update` | Service (`X-API-Key`) | Updates a node's state; the processing layer invokes it, and the backend broadcasts the change over WebSocket. |

### Custom queries (`/custom-queries`)

| Method | Route | Auth | Description |
| --- | --- | --- | --- |
| GET | `/custom-queries/ultimas-alertas` | User | Latest recorded alerts. |
| GET | `/custom-queries/total-alertas` | User | Total number of alerts. |
| GET | `/custom-queries/alertas-por-nodo` | User | Alerts grouped by node. |
| GET | `/custom-queries/canales-mas-afectados` | User | Channels with the most alerts. |
| GET | `/custom-queries/alertas-de-hoy` | User | Today's alerts. |
| GET | `/custom-queries/alertas-por-fecha` | User | Alerts filtered by date. |

### Logs and administration

| Method | Route | Auth | Description |
| --- | --- | --- | --- |
| GET | `/logs` | Administrator | Lists the log files. |
| GET | `/logs/{log_filename}` | Administrator | Returns the contents of a log. |
| GET | `/logs/download/{log_filename}` | Administrator | Downloads the complete log file. |
| GET | `/admin/logs` | Administrator | Log view for administration. |
| GET | `/admin/reports` | Administrator | Report view for administration. |

### WebSocket

- **URL:** `ws://<host>:<port>/ws/alerts?token=<JWT>` (for example, `ws://localhost:8000/ws/alerts?token=...`).
- The JWT travels as a query parameter, because the browser WebSocket does not support custom headers in the *handshake*. Without a valid token, the connection is rejected with close code 1008 before it is accepted.
- On a deployment exposed to the Internet, use WSS to encrypt the token in transit.
- It broadcasts new alerts and node state changes in real time.

## Models and schemas

| File | Engine | Contents | Description |
| --- | --- | --- | --- |
| `models.py` | SQLAlchemy | `Alert`, `ESP32Status` | `alerts` and `esp32_status` tables, with their fields and constraints. |
| `schemas.py` | Pydantic | `AlertCreate`, `AlertResponse`, `ESP32StatusResponse` | Validation and serialization of requests and responses. |

## Database

PostgreSQL is used. The tables are created automatically when the application starts (`Base.metadata.create_all`):

| Table | Description |
| --- | --- |
| `alerts` | Deauthentication events (BSSID, channel, date). |
| `esp32_status` | State (`connected`/`disconnected`) of each node. |
| `users` | Dashboard user accounts (created by the backend itself). |
